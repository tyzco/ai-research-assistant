## 一、RAG.md 提到的优化 — 逐项对照

### 1. 文档切分粒度

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 500-1000 字符，带部分 overlap | `semantic_chunk_text()` 按句子边界切分，`max_chars=1024`，overlap=2 句 | ✅ 已做 |
| 太粗会混入无关信息，太细会丢失上下文 | `chunk_text()` 固定 2048 字符 + 100 字符 overlap（滑窗备选） | ✅ 已做 |
| — | 双层索引：摘要层（覆盖广度）+ 全文层（深度细节） | ✅ **超出** |
| — | 最小块过滤（<50 字丢弃） | ✅ **超出** |

**评价**：RAG.md 建议的"语义分块 + overlap"已实现。额外做了双层索引和最小块过滤。未做的是"父子分块"（parent-child chunking）——检索小 chunk、返回大 context 的策略，这在长文档场景下可以进一步提升上下文完整性。

### 2. 检索不相关

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 换更好的 embedding 模型 | 实测 `bge-large-zh`（73.6%）反不如 `bge-small-zh`（81.6%），保留 small | ✅ 已做（有消融验证） |
| Hybrid search（向量 + 关键词，如 Elasticsearch） | LanceDB FTS（BM25）+ 向量，RRF 融合 | ✅ 已做 |
| — | **自适应 RRF**：4 维特征动态调整 BM25/向量权重，非固定 50:50 | ✅ **核心创新** |
| — | 8x 宽召回（top_k × 4 候选），给重排留空间 | ✅ **超出** |
| — | 三层检索（全文 + 图片 + 摘要），各层独立混合检索 | ✅ **超出** |

**评价**：这部分的实现**全面超越**了 RAG.md 的建议。RAG.md 只提到"换模型、加 BM25"，项目实现了自适应融合权重、宽召回、三层架构。自研的 Adaptive RRF 比 LangChain/LlamaIndex 的固定权重方案更灵活。

**为什么没选 Elasticsearch？**
- LanceDB 嵌入式部署，零运维，适合单机场景
- ES 需要 Java 运行时 + 独立进程，对个人项目过重
- LanceDB 原生支持向量搜索 + FTS + 元数据过滤，三合一

**消融数据支持决策**：消融实验显示 Hybrid+QueryExp Top-10（0.811）仅比纯向量 Top-5（0.8081）高 0.3%。这说明 bge-small-zh 的语义表达已经很好，BM25 补充的边际收益有限，但自适应权重在特定查询类型（短关键词、缩写术语）上有明显帮助。

### 3. 丢信息（top-K 只取 3 段，答案分散在 5 段）

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 增大 K | `RETRIEVAL_TOP_K=5`，宽召回取 top_k × 4 = 20 候选 | ✅ 已做 |
| 先检索后做重排（Rerank），如 Cohere Rerank | **两阶段级联重排**：MiniLM 预筛 → bge-reranker 精排 | ✅ 已做 |

**与 Cohere Rerank 的对比选择**：
- Cohere Rerank：云端 API，按次付费，有网络延迟
- bge-reranker-base：本地 CrossEncoder，~1.3GB，零成本，无网络依赖
- 选择 bge 的理由：与 bge-small-zh 嵌入同属 BAAI 系列，中文优化，且免费

**级联设计的额外收益**：Stage1 用轻量 MiniLM 过滤掉明显不相关的，Stage2 只在 15 个候选上跑 CrossEncoder，比直接对所有候选做重排节省约 40% 时间。

### 4. 找不到答案时模型还硬答

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 兜底逻辑：相似度低于阈值就回答"资料里没有" | QA_PROMPT 第 5 条规则："如果信息不足请明确说明" | ✅ 已做 |
| — | Prompt 区分"全文来源"和"补充线索"，模型知道哪些是可靠的 | ✅ **超出** |
| — | 诚实奖励：prompt 中强调"准确比全面更重要" | ✅ **超出** |

**效果验证**：Faithfulness 从 0.40 → 0.57 → 0.82（v1→v3→v6），证明 Prompt 约束 + 检索质量提升的双重作用。

**为什么没有用相似度阈值做硬拦截？**
- 余弦相似度阈值很难调——不同查询的"相关"阈值不同
- 硬拦截可能误杀"部分相关但仍可回答"的查询
- 选择 soft 约束（Prompt 拒答机制）更鲁棒，让 LLM 自己判断

### 5. 多轮对话

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 每次检索时带上历史对话摘要 | `ConversationMemory`：滑动窗口 5 轮 + 滚动摘要，注入 Prompt | ✅ 已做 |
| 把历史放进检索 Query | 未实现（检索只用当前问题） | ❌ 缺失 |

**缺失项分析**：RAG.md 提到"把历史也放进检索 Query"——这指的是 **Contextualized Query Rewriting**，即将对话历史融入检索查询。例如：
- 用户 Q1："什么是 RAG？"
- 用户 Q2："它有什么缺点？"
- 理想检索 Query："RAG 的缺点"而非"它有什么缺点"

当前项目只做了 Query Expansion（同义词扩展），没有做多轮指代消解。这是一个**值得补的优化**，用 DeepSeek 做一轮 Historical Query Rewriting 即可（约 2-3s 额外延迟）。

### 6. 成本与延迟

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 用更小的 embedding 模型 | bge-small-zh（512d，~100MB，CPU 0.1s） | ✅ 已做 |
| 缓存常见问题 | 未实现 | ❌ 缺失 |
| — | 条件查询改写（短问题跳过） | ✅ **超出** |
| — | 条件重排（简单问题跳过） | ✅ **超出** |
| — | Agent 快慢分离（quick 3s vs ReAct 30s） | ✅ **超出** |
| — | 嵌入模型启动时预加载（首次请求无需等待） | ✅ **超出** |

**缺失的缓存分析**：对高频学术问题做语义缓存（如 GPTCache 或 Redis）可以进一步提升响应速度。但学术场景的查询长尾分布明显，缓存命中率可能不高，收益有限。

**延迟优化效果**：
- 创建课题：17s → 6s（前端拆分串行等待）
- Agent 问答：30s → 3-5s（快慢分离）
- 嵌入：6-8min（Jina API 云端）→ 0.1s（bge 本地）

### 7. 私有数据安全

| RAG.md 建议 | 项目实现 | 状态 |
|------------|---------|------|
| 本地部署 embedding（如 BGE） | bge-small-zh 本地 CPU 推理 | ✅ 已做 |
| 本地 LLM（如 DeepSeek） | DeepSeek API（云端），非本地 | ⚠️ 半实现 |
| 数据不出内网 | 嵌入在本地，但生成调 DeepSeek API | ⚠️ 半实现 |

**为什么没做全本地 LLM？**
- DeepSeek API 质量远好于本地小模型，且中文极强
- 本地跑 7B 模型（如 Qwen2.5-7B）需要 >8GB GPU 显存，不符合 2C4G 无 GPU 的部署目标
- 权衡：嵌入层本地化（数据不出内网）+ 生成层云端（用最好的模型），是当前最优折中

## 二、RAG 演进阶段对照

RAG.md 第十一节定义了 4 个演进阶段：

| 阶段 | 特征 | 项目实现情况 |
|------|------|-------------|
| **Naive RAG** | 固定分块 → 向量检索 → 生成 | ✅ 已超越 |
| **Advanced RAG** | 混合检索 + 重排序 + 查询重写 | ✅ **完整实现** |
| **Modular/Agentic RAG** | 自主规划、反思纠错、多源检索、工具调用 | ✅ **部分实现**（ReAct Agent + 8 工具，但无反思机制） |
| **知识编译 (Knowledge Compilation)** | 预编译知识，结构化存储，声明式查询 | ❌ 未实现 |

### Advanced RAG — 完整覆盖

项目的 Advanced RAG 实现程度：

| 子项 | 是否实现 | 成熟度 |
|------|---------|--------|
| 混合检索（向量 + BM25） | ✅ | 高（自适应 RRF） |
| 重排序 | ✅ | 高（两阶段级联） |
| 查询重写 | ✅ | 高（DeepSeek + 语义校验） |
| 多路召回 | ✅ | 高（6 源并行搜索） |
| 三层检索架构 | ✅ | 高（全文 + 图片 + 摘要） |

### Agentic RAG — 有基础但缺少反思

| 子项 | 是否实现 | 说明 |
|------|---------|------|
| 自主规划 | ✅ | ReAct 循环，LLM 自己决定下一步用什么工具 |
| 多源检索 | ✅ | 6 个学术数据源，8 个 Agent 工具 |
| 工具调用 | ✅ | OpenAI Function Calling 标准格式 |
| 反思纠错 | ❌ | **没有** Self-Reflection 步骤 |
| 多 Agent 协作 | ❌ | **没有** Planner/Searcher/Analyst 分工 |
| LangGraph 图状工作流 | ❌ | 代码脚手架存在但未集成 |

**为什么没做反思（Reflection）？**
- ReAct 循环本身有隐式纠错（工具返回错误 → LLM 看到 → 调整下一步）
- 显式 Reflection 会增加 1-2 步额外的 LLM 调用（每次 ~8s），显著增加延迟
- 当前 10 步上限的 ReAct 在简单任务上已经够用
- 复杂调研任务确实需要 Reflection，这是一个**明确的下一步**

**为什么不追 LangGraph/多 Agent？**
- RAG.md 自己也说："先从 Advanced RAG 的工程优化开始，通常能解决 80% 以上的问题，不要一上来就追 Agentic RAG 这个银弹，成本会高很多"
- 项目遵循了这个建议——先把检索质量做到 82%，再逐步加 Agent 能力

## 三、RAG.md 没提到但项目做了的事（额外加分项）

### 3.1 多模态 RAG

RAG.md 完全没有涉及图片/图表的处理。项目实现了：
- PDF → PyMuPDF 提取图片 → 规则过滤（尺寸/长宽比/颜色数/页码） → Qwen-VL 生成中文描述 → 向量化存入 LanceDB → 检索时匹配

这是 RAG.md 知识体系之外的完整多模态流水线。

### 3.2 多源学术搜索引擎

RAG.md 假设"知识库就在那里"，没讨论知识库从哪来。项目自建了完整的学术论文发现系统：
- 6 源并行搜索（arXiv + OpenAlex + Semantic Scholar + Unpaywall + CNKI + Apify）
- 14 条垃圾过滤规则（征稿、广告、撤稿等）
- 来源权重排序 + 年份衰减
- 中文论文两阶段相关性过滤
- Unpaywall OA PDF 自动补链

### 3.3 全链路 Graceful Degradation

RAG.md 提到"兜底逻辑"，但项目实现了更系统的降级策略：

| 失败层级 | 降级行为 |
|---------|---------|
| DeepSeek 查询改写失败 | 用原始查询 |
| 语义校验不通过 | 拒绝改写，用原始查询 |
| AdaptiveHybridRetriever 异常 | 回退到简单 `retrieve_with_images()` |
| bge-reranker 加载失败 | 用 MiniLM 结果截断 |
| MiniLM 也失败 | 用原始向量分排序 |
| LLM 超时 | `safe_call` 重试 2 次 + 返回 None → 友好错误消息 |
| PDF 魔数校验失败 | 拒绝上传，防止恶意文件 |

### 3.4 安全三层防护

RAG.md 只提了"私有数据安全"一个方面。项目额外做了：
1. **输入层**：15 正则模式 prompt 注入检测（中英文），控制字符过滤，长度截断
2. **文件层**：PDF 魔数校验，路径遍历防护
3. **监控层**：MetricsTracker 全链路计数 + `/metrics` 端点

### 3.5 评估体系（无重型依赖）

RAG.md 完全没有涉及如何评估 RAG 质量。项目自建了评估系统：
- 8 条标注 QA 对（人脸识别领域）
- 3 项指标：Faithfulness（LLM-as-judge）、Context Precision（嵌入余弦）、Semantic Recall@5
- 消融实验：纯向量 vs 混合 vs 混合+改写，Top-5 vs Top-10
- 15 题 × 50 篇论文 × 5622 块的标准化评估
- 嵌入模型对比基准：bge-large vs bge-small

### 3.6 前端交互优化

- 论文中英文分组显示（🇨🇳/🌐）
- Agent 步骤可视化卡片
- 策略展示（布尔检索式 + 数据库 + 关键词）
- 一键下载并构建知识库 + 实时进度条
- 模型选择器（LLM + Vision 独立切换）
- 暗色模式

### 3.7 认证与多租户

RAG.md 从未讨论。项目实现了 JWT + bcrypt 用户认证，JSON 文件持久化，匿名访问模式。

## 四、方法选择决策树 — 为什么选了 A 而不是 B

### 4.1 为什么 LanceDB 而不是 Chroma/Pinecone/Milvus？

| 方案 | 优点 | 为什么不选 |
|------|------|-----------|
| Chroma | Python 原生，易上手 | 无 BM25 全文检索，大 KB 性能下降 |
| Pinecone | 全托管，性能好 | 收费，需网络，数据出内网 |
| Milvus | 功能最全，分布式 | 需独立部署，太重（2C4G 扛不住） |
| Weaviate | 混合搜索原生支持 | 需 Java 运行时 |
| **LanceDB** ✅ | 嵌入式，向量+BM25+元数据过滤，列式存储 | 社区小，API 变动频繁，但功能恰好满足需求 |

### 4.2 为什么 bge-small-zh 而不是 text-embedding-3-small / bge-m3？

| 方案 | 为什么不选 |
|------|-----------|
| OpenAI text-embedding-3-small | 付费 API，中文不如 BGE，数据出内网 |
| bge-large-zh-v1.5 (1024d) | **实测更差**（73.6% vs 81.6%），MTEB 确认大模型在 sentence-level 劣于 small |
| bge-m3 (1024d, 多语言) | 未测试。可能更好（多语言 + 8192 token 上下文），但模型更大（~2GB），收益不确定 |
| **bge-small-zh** ✅ | 免费、本地、快、实测最优 |

### 4.3 为什么自研 Adaptive RRF 而不是用 LangChain 的 EnsembleRetriever？

LangChain 的 `EnsembleRetriever` 使用固定权重 RRF（k=60，等权）。本项目的 Adaptive RRF 在此基础上做了 4 维特征驱动的动态权重。

**消融数据显示增益很小（0.808→0.811）**，理论上不值得投入。但价值在于：
1. 证明了"动态权重 > 固定权重"的概念可行性
2. 在查询分布更广的场景（如包含大量缩写、短关键词的专业查询）下，自适应权重收益会放大
3. 面试时可展示独立思考和工程能力

### 4.4 为什么没有用 LangChain/LlamaIndex 做编排？

| 方案 | 优点 | 为什么不选 |
|------|------|-----------|
| LangChain | 生态最大，组件最全 | 依赖树巨大（200+ 包），版本冲突频繁（亲身经历 langchain_community 安装冲突）；高层抽象屏蔽了 RAG 内部细节，学不到东西 |
| LlamaIndex | 专注 RAG，索引策略丰富 | 同样重依赖；其混合检索是固定权重 |
| **自研** ✅ | 完全控制，零冗余依赖 | 维护成本高，但 16 个文件、~3000 行代码可控 |

"对于个人项目和快速迭代，自研更灵活。团队协作的生产项目用 LangChain 生态更好。"（来自项目总结）

### 4.5 为什么没有做 Graph RAG？

RAG.md 提到微软的 Graph RAG 可以解决多跳推理问题。项目目前没有做，因为：
- Graph RAG 需要实体识别 + 关系抽取 + 图构建 + 图查询，工程量大
- 当前场景（学术文献问答）以单跳问题为主（"XX 论文用了什么方法？"），多跳需求少
- 如果用户需要"A 论文引用了谁，谁又引用了 B"这种引用链查询，Graph RAG 才有明显优势
- **可以作为一个扩展方向**：用知识图谱表示论文引用关系

## 八、LongRAG 大块检索实验 (2024-06-21)

### 实验设计

| 参数 | 基线 (v1.2) | LongRAG | 
|------|-----------|---------|
| chunk_size | 2048 chars | 4096 chars |
| chunk 数量 (50 papers) | 5622 | 1065 |
| chunk 数减少 | — | **81%** |

### 结果

| 指标 | 基线 2048 | LongRAG 4096 | 变化 |
|------|----------|-------------|------|
| Semantic Recall@5 | 0.7719 | 0.7737 | +0.0018 |
| Context Precision | 0.7556 | 0.7536 | -0.0020 |
| ≥0.80 通过率 | 2/8 | 1/8 | -1 |

### 分析

**检索指标基本持平**——增大 chunk 不会提升检索召回率，因为 embedding 被更长的文本稀释，向量表达的聚焦度下降。但 LongRAG 的真正价值在**生成阶段**：LLM 每个 chunk 获得 2 倍上下文，减少"缺表头表格""缺 if 条件的论证"等语义断裂。

**何时切换**：
- 当前 2048 → 适合句子级检索，语义精确
- 切换 4096 → 适合长窗口 LLM（DeepSeek 128K），减少 chunk 数降低检索延迟
- 下一步可尝试 **4K token 级**（约 12000 字符），利用现代 LLM 的 128K 上下文窗口直接塞入更多完整段落

### 面试话术

> "我跑了一个 LongRAG 的对照实验——把 chunk 从 2048 扩大到 4096。检索召回指标基本持平，但 chunk 数量减少了 81%，这在实际部署中能显著降低向量库体积和检索延迟。LongRAG 的真正收益在生成阶段——LLM 看到更完整的上下文——这需要做生成质量的人评才能量化。"

## 九、前沿方向技术视野（未实现，文档展示用）

### 9.1 GraphRAG — 实体关系图谱增强检索

**是什么**：微软提出的方法，从文档中抽取实体（方法名、数据集、指标）和关系（SOTA、优于、基于），构建知识图谱。查询时在图谱中做多跳推理。

**我们的现状**：当前项目基于向量相似度检索，无法回答"哪些方法在 ImageNet 上准确率超过 90%？"这类关系推理问题。

**实现路径**（如果要做的优先级顺序）：
1. 用 DeepSeek 从摘要中抽取 `(方法, 数据集, 指标值)` 三元组
2. 存为 JSON/Neo4j，查询时用 Cypher
3. 检索结果与向量检索结果融合排序

**为什么暂时不做**：需要完整的信息抽取管线 + 图数据库，工程量相当于一个独立项目。当前学术文献问答以单跳查询为主，GraphRAG 的边际收益有限。

### 9.2 REFRAG — Token 级压缩检索

**是什么**：用轻量编码器将 16 个 token 压缩为 1 个向量，检索后在嵌入空间直接做推理，减少主 LLM 的输入 token 数。

**我们的现状**：当前每个 chunk 512-2048 字符直接送入 LLM Prompt，未做压缩。

**为什么暂时不做**：
- 需要额外训练或微调一个压缩编码器
- 强化学习挑选关键片段需要标注数据或奖励设计
- 学术论文单篇不长（~30K 字符），当前 context 还没到瓶颈
- 投入产出比远低于工程优化（检索质量、分块策略、Prompt 设计）

### 9.3 如何将这些展示为技术视野

在面试和 README 中，可以这样说：

> "目前我们已经实现了流程智能化——自适应路由、动态混合检索权重、级联重排、全链路降级。对于更前沿的技术，我在技术文档中预留了分析和扩展路径：
> - **LongRAG**：已跑对照实验，chunk 从 2048→4096，chunk 数减少 81%，检索指标持平。生成质量的提升需要人评验证。
> - **GraphRAG**：分析了适用场景——当查询需要跨论文的关系推理时（'哪些方法在 XX 数据集上超越基准'），需要引入实体关系图谱。我已经规划了三元组抽取→图构建→混合检索的实现路径。
> - **REFRAG**：理解了 token 压缩的核心思想，但在当前规模下（50-100 篇论文），直接全文检索的延迟和成本都可接受，暂时不需要压缩。
>
> 我的原则是：**先做工程优化（解决 80% 问题），再追方法论创新**。这也是我在改进评估中得到的经验——简单地扩大 chunk 不会提升检索，需要理解底层 embedding 的工作原理。"
