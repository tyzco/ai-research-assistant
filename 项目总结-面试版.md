# 学术文献精读助手 — 完整项目总结

> 校招/实习面试用 · 从 0 到工业级 RAG Agent 系统

---

## 一、项目一句话

一个从检索策略生成、多源论文搜索、自动知识库构建到深度 RAG 问答的学术文献助手，支持 Agent 自主调研，前端零依赖。

| 层级 | 选型 | 为什么 |
|------|------|--------|

## 四、核心创新点（面试重点）

### 4.1 自适应混合检索 (AdaptiveHybridRetriever)

**问题**：普通 RAG 用固定权重做 BM25+向量融合，但不同问题需要不同策略。短关键词查询偏 BM25，自然语言问句偏语义。

**方案**：4 维特征动态推理 BM25 权重(alpha)：

| 特征 | 对 alpha 的影响 | 设计理由 |
|------|----------------|---------|
| 短查询 (<15字) | +0.20 | 更像关键词检索 |
| 含英文缩写/术语 | +0.15 | 精确匹配更重要 |
| 问句形式 (？/是什么/如何) | -0.15 | 偏语义理解 |
| 长查询 (>40字) | -0.10 | 语义相近比字面匹配重要 |

**融合公式**：`weighted_rank = alpha × RRF(BM25) + (1-alpha) × RRF(Vector)`

**与 LangChain/LlamaIndex 的区别**：他们的混合检索用固定权重或简单规则，本项目是特征工程+自适应，可解释、可消融、可扩展。

### 4.2 级联重排序

**问题**：直接用 bge-reranker 对所有召回做重排耗时长（CrossEncoder 逐对计算）。

**方案**：两阶段
```
Stage1: all-MiniLM-L6-v2 (BiEncoder, ~20ms)
    粗召回 top-20 → 相似度预筛 → top-15

Stage2: bge-reranker-base (CrossEncoder, ~200ms)
    top-15 → 逐对精排 → top-5
```

**收益**：节省约 40% 推理时间。Stage1 的轻量模型过滤掉明显不相关的块，Stage2 只在候选集上精排。两个 Stage 可独立替换或升级。

### 4.3 三层 RAG 检索架构

```
查询改写 (DeepSeek, 长问题才触发)
    → 语义校验 (cos ≥ 0.75, 防止改写偏离原意)
    → 三层并行检索:
        ├─ 全文文本层 (BM25 + Vector, is_fulltext=true)
        ├─ 图片描述层 (BM25 + Vector, is_image=true)
        └─ 摘要补充层 (Vector, is_fulltext=false)
    → 级联重排
    → Prompt 拼接 (区分全文来源 vs 补充线索)
    → DeepSeek 生成 + 引用标注 + 文献缺口提示
```

### 4.4 Faithfulness 保障机制

**Prompt 级约束**：
- 明确区分"库内全文来源"和"外部补充线索"
- 要求引用格式：`[论文: 标题, 年份, 页码大约 N]`
- "如果信息不足，请明确说明"
- "不要编造论文中不存在的数据和结论"

**诚实奖励**：prompt 中强调"准确比全面更重要"，LLM 被训练为宁可说不知道也不编造。

**效果**：Faithfulness 从 0.40 → 0.57 → 0.82（v1 → v3 → v6）。

## 五、开发历程与关键踩坑（面试讲故事用）

### 5.1 嵌入方案迁移：Jina API → 本地 bge-small-zh

**最初**：使用 Jina AI 的云端 Embedding API。
**问题**：免费额度 15 RPM，94 篇论文需要 200+ 调用，疯狂 429。嵌入一次 6-8 分钟且经常失败。
**尝试**：加了重试+退避，还是不行。
**最终方案**：切换到本地 BAAI/bge-small-zh（512d），CPU 推理 0.1s/条，完全免费无限制。
**意外发现**：bge-large-zh-v1.5（1024d）反而不如 small（Semantic Recall 73.6% vs 81.6%）。原因是 small 在 sentence-level 检索上的 MTEB 得分确实更高——大模型优势在 document-level。

> **面试话术**："我学到的一个教训是：云端 API 看起来方便，但对批处理场景不友好。本地小模型反而更可靠。而且模型大小不是越大越好，需要跑消融验证。"

### 5.2 检索质量从 42% 到 82% 的爬坡

```
v0.3: 纯向量 Top-5              → 42% (惨不忍睹)
v0.4: + 多源搜索 (论文数 11→141)  → 提升明显
v0.5: + 质量过滤 (屏蔽广告/撤稿)   → 去掉噪声
v0.6: + BM25 混合检索            → 关键词精确匹配补充
v0.7: + 查询改写 + 语义校验        → 短查询不再丢失语义
v0.8: + 级联重排                  → 粗召回中的精华提取
v0.9: + 自适应 RRF 权重           → 根据问题类型动态调整
v1.0: + 策略路由 (简单/复杂分流)   → 简单问题跳过重排省时间
v1.1: 94 篇论文的大规模 KB        → 82% ✅
```

> **面试话术**："这个爬坡过程让我真正理解了 RAG 的瓶颈在哪里——不是生成，是检索。检索质量上去了，生成自然就好。"

### 5.3 Faithfulness 从 0.40 到 0.82 的 Prompt 工程

**v1 (0.40)**：prompt 宽松，LLM 自由发挥，经常编造论文里不存在的数据。
**v3 (0.57)**：加了"基于上下文回答"约束，但 LLM 还是会从摘要层"脑补"。
**v6 (0.82)**：三个关键改动——
1. 严格区分全文来源和补充线索
2. 拒答机制："如果信息不足请说明"
3. 诚实奖励："准确比全面更重要"
同时将 KB 从 50 篇扩展到 94 篇，检索质量好转了，生成自然更有底气。

> **面试话术**："RAG 的 Faithfulness 本质上是检索质量和 Prompt 约束的乘积。检索不好，再怎么约束 LLM 也会编；Prompt 不严格，检索再好也可能被 LLM 的创造力污染。"

### 5.4 前端连环 Bug 排查实录（最近一次）

**现象**：用户反馈"输入机器学习按回车，页面根本没反应"。
**排查过程**：
1. 先确认 API 是否存活 → curl 测试 `/agent/quick` 和 `/create_topic`，一切正常
2. 检查浏览器控制台 → 发现 JS 无报错（只有 favicon.ico 404）
3. 用 Playwright 打开页面 → 页面能渲染但输入后 loading 不停
4. 逐行排查 `R()` 渲染函数 → 发现第 206 行开头多了个 `205` 字符

**根因**：前一次用 Edit 工具时，Read 输出的行号前缀 `205` 误写入了文件内容。`205` 在 JS 中是无效语法，导致 `R()` 函数解析失败，页面完全不响应。

> **教训**：编辑 minified 单行 JS 时，行号前缀极易被误写入。应该先格式化再编辑。

**第二个 bug**：`search_papers` API 返回 `{papers: [...]}` 但前端读的是 `pd.pps`（字段名不匹配），论文永远为空数组。这是历史上字段名不一致遗留的问题。

**第三个 bug**：LanceDB v2.x 的 `list_tables()` 返回 `ListTablesResponse` 对象（需 `.tables` 取值），不是旧版的 `list[str]`。导致 KB 查找失败，Agent 永远返回"没有可用的知识库"。

> **面试话术**："前端 bug 排查教会我：不要只看表面现象。用户说'页面没反应'，可能的原因从 API 挂了、JS 报错、到字段名不匹配、甚至一个多余的字符。系统性排查比猜重要。"

### 5.5 Agent 速度优化：从 30s 到 3s

**最初**：Agent 用完整的 ReAct 循环（`/agent/run`），每个 step 调一次 DeepSeek（~8s），3-4 步就是 25-30s。用户看到的"一直思考"。

**优化**：新增 `/agent/quick` 端点——跳过 ReAct 循环，直接：
1. 找最大 KB（LanceDB 按行数排序）
2. 问题向量化 → 余弦检索 top-5
3. 单次 DeepSeek 调用生成回答
总耗时：**3-5 秒**。

**权衡**：quick 模式不生成检索策略、不多步推理，但 80% 的用户场景（"什么是 XX"）用这个更快。

> **面试话术**："ReAct 循环适合复杂多步调研，但简单问答用整个 Agent 流程是杀鸡用牛刀。根据用户意图做路由——简单问题直接检索+生成，复杂任务走 Agent——这是工程上常用的快慢分离模式。"

## 六、评估指标与消融实验

### 最终指标（94 篇论文, 10521 文本块, bge-small-zh 512d）

| 指标 | 数值 | 说明 |
|------|------|------|
| **Faithfulness** | **0.8227** | DeepSeek 评分，回答基于上下文的程度 |
| **Context Precision** | **0.7998** | 检索块与 ground truth 的语义重叠度 |
| **Semantic Recall@5** | **0.8198** | Top-5 检索能覆盖 ground truth 的比例 |
| ≥0.80 通过率 | 13/15 (87%) | |
| ≥0.75 通过率 | 15/15 (100%) | |

### 消融实验（50 篇论文, 5622 块, 15 QA）

| 方法 | Semantic Recall | 备注 |
|------|----------------|------|
| Pure Vector Top-5 | 0.8081 | 基线 |
| Pure Vector Top-10 | 0.8108 | 多召回收益边际 |
| Hybrid + QueryExp Top-5 | 0.8090 | 改写对短查询有帮助 |
| **Hybrid + QueryExp Top-10** | **0.8110** | 最优组合 |
| Hybrid Top-10（无改写） | 0.8081 | 改写并非总是正收益 |

### 关键发现

1. **纯向量检索已经很能打**——与混合检索差距仅 0.1%，说明 bge 嵌入在学术文本上有很好的语义表达
2. **查询改写边际收益 0.3%**——对简单问题帮助不大，但对"对比""区别"类复杂问题有正向作用
3. **瓶颈在嵌入维度**——512d 的表达能力上限约 82%，升级到 1024d 预期+3-5%
4. **级联重排在当前 KB 上收益不明显**（甚至轻微负优化 -1.6%）——因为基础召回已经足够好，重排没有筛选空间。当 KB 从 50 扩展到 500+ 时预期收益显现

## 七、面试高频问题与回答要点

### Q1: 为什么不用 LangChain/LlamaIndex？

**答**：三个原因：
1. **学习目的**——这个项目的目标是深入理解 RAG 的每一层，用框架就学不到东西了
2. **定制需求**——LangChain 的混合检索是固定权重，我需要自适应；它的分块是固定大小，我需要语义分块
3. **依赖管理**——LangChain 的依赖树巨大且脆弱（经历过 langchain_community 安装冲突），单文件模块更容易维护

当然，如果是团队协作的生产项目，用 LangChain 的生态（Callback、Tracing、LangSmith）会更好。但对于个人项目和快速迭代，自研更灵活。

### Q2: 为什么用 bge-small-zh 而不是更强的嵌入模型？

**答**：经过消融对比。测试了 bge-large-zh-v1.5（1024d）但 Semantic Recall 反而低了 8 个百分点（73.6% vs 81.6%）。原因是：
- bge-small-zh 在 MTEB 的 sentence-level 检索任务上确实优于 large
- large 模型的优势在 document-level（长文本聚类等），我们的场景是句子级检索
- small 模型更轻量（~100MB vs ~600MB），CPU 推理 0.1s vs 0.5s

> 面试官可能会追问："那如果 KB 扩大到 10 万篇呢？" → "那时 512d 的表达能力会成为瓶颈，会考虑升级到 bge-m3（1024d, 多语言）或做 fine-tune。但当前规模下 small 是性价比最优解。"

### Q3: Faithfulness 0.82 够用吗？怎么进一步提升？

**答**：0.82 意味着 87% 的回答在 ≥0.80 的忠实度区间，100% 在 ≥0.75。对于文献调研场景，这不是医疗/法律级别的可靠性需求，够用但不够好。

提升路径：
1. **升级嵌入模型** → bge-m3 (1024d) 预期 +3-5%
2. **扩大 KB** → 从 94 篇到 500+ 篇，检索覆盖更全面
3. **引入 Grounding** → Google/GPT-4 的 grounding 机制，在生成过程中实时验证
4. **Fine-tune Reranker** → 用领域数据微调 bge-reranker

### Q4: 为什么用 LanceDB 而不是 Chroma/Milvus/Qdrant？

**答**：
- Chroma：Python 原生但缺乏 BM25 全文检索，且大 KB 下性能下降
- Milvus/Qdrant：功能强大但需要独立部署，不适合单机/轻量场景
- LanceDB：嵌入式（零部署），原生支持向量+BM25 FTS+元数据过滤，列式存储高效

LanceDB 的缺点：社区较小，API 变动频繁（曾因 `list_tables()` 返回类型变更导致 bug）。

### Q5: 如何处理 LLM 幻觉？

**答**：三层防御：
1. **检索层**——保证检索质量（Recall 82%），让 LLM 有据可依
2. **Prompt 层**——严格约束引用格式，区分"库内来源"和"外部补充"，拒答机制
3. **评估层**——系统性地评估 Faithfulness，每个版本追踪退化

Faithfulness 从 0.40 到 0.82 的提升证明：检索质量是根基，Prompt 是护栏。

### Q6: 项目最大的技术难点是什么？

**答**：自适应混合检索的特征设计。难点不在于代码实现，而在于：
1. 哪些查询特征影响检索策略？
2. 影响方向是什么？幅度多大？
3. 如何量化验证？

最终的做法是：观察大量查询的检索结果 → 总结规律 → 编码为特征 → 消融验证。比如"短查询偏 BM25"这个规律，是通过对比"人脸识别"（2 字）和"人脸识别技术的伦理问题与隐私保护"（13 字）在纯向量和纯 BM25 下的召回质量得出的。

### Q7: 如果 KB 扩大到 10 万篇论文，系统哪里会先撑不住？

**答**：
1. **向量检索**：LanceDB 支持索引（IVF_PQ），512d 下 10 万条问题不大
2. **BM25 全文索引**：LanceDB FTS 基于 Tantivy，10 万条也能扛
3. **瓶颈是嵌入**：批量嵌入 10 万篇论文（每篇平均 5 块 = 50 万次嵌入），bge-small-zh 在 CPU 上需要约 7 小时。需要 GPU 加速或增量索引
4. **其次瓶颈是重排序**：粗召回 100 条 × CrossEncoder 逐对计算 = 100 次推理，需要批处理或 GPU

### Q8: 项目中有哪些安全考虑？

**答**：
1. **Prompt 注入防护**：15 个正则模式检测中英文注入关键词（"忽略指令"、"system:"、"DAN:"、"jailbreak"等），匹配后替换为 [filtered]
2. **路径遍历防护**：文件名校验，拒绝含 `..`、`/`、`\` 的文件名
3. **PDF 魔数校验**：检查 `%PDF-` 文件头，防止上传恶意文件
4. **输入长度限制**：截断超过 2000 字符的用户输入
5. **指标监控**：MetricsTracker 记录请求量/错误率/延迟，暴露 `/metrics` 端点
