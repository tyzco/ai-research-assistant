## 二、各文件详解

### 1. `config.py` — 全局配置中心

**功能**：所有 API Key、路径、模型参数统一管理。

**关键配置**：
```python
DEEPSEEK_API_KEY / DEEPSEEK_MODEL    # 生成 LLM
EMBEDDING_PROVIDER / LOCAL_EMBEDDING_MODEL  # 嵌入模型 (bge-small-zh)
VISION_MODEL / VISION_API_KEY        # 视觉模型 (可选)
SEMANTIC_SCHOLAR_API_KEY / APIFY_API_KEY  # 学术搜索 Key
```

**依赖**：无（被所有模块依赖）

**建设顺序**：第 1 个文件。项目一开始就确定所有配置项，后续模块通过 `from config import ...` 引用。

---

### 2. `models.py` — Pydantic 数据模型

**功能**：定义所有数据结构。

**核心类**：
- `PaperMeta` — 论文元数据 (paper_id, title, authors, year, doi, pdf_url, is_oa)
- `TopicState` — 课题运行时状态 (status, progress, lancedb_table)
- `AskRequest / AskResponse` — 问答 API 的请求/响应格式
- `TopicStatus` — 课题状态枚举

**依赖**：无

**建设顺序**：第 2 个文件。数据结构先行，后续所有模块都基于它。

---

### 3. `search.py` — 多源论文搜索

**功能**：用户输入研究方向 → LLM 生成检索策略 → 多源并行搜索论文。

**核心函数**：
- `generate_search_strategy(query)` — DeepSeek 生成：关键词 + 领域映射 + 检索式 + 关键学者
- `search_papers_for_topic(query, keywords_en, keywords_cn)` — 并行搜索主入口
- `_search_s2()` — Semantic Scholar API
- `_search_arxiv()` — arXiv API
- `_search_openalex()` — OpenAlex API (英文)
- `_search_openalex_cn()` — OpenAlex API (中文, language:zh 过滤)
- `_search_google_scholar_apify()` — Apify 谷歌学术 (需 Key)
- `_make_cnki_async()` — 知网搜索链接生成
- `_enrich_via_unpaywall()` — Unpaywall DOI → PDF 补链
- `_detect_cn()` — charCode 中文检测
- `_cn_relevant()` — 中文论文关键词相关性过滤
- `_quality_score()` — 来源权重评分
- `_should_drop()` — 垃圾论文过滤 (广告/撤稿/征稿)

**优化历程**：
1. 最初只有 Semantic Scholar + arXiv
2. 加入 OpenAlex (并行搜索，论文数 11→141)
3. 加入 OpenAlex CN 过滤 (中文 7→37)
4. 加入 Unpaywall 补链
5. 加入质量过滤 + 屏蔽关键词
6. 加入 Apify 谷歌学术
7. 放宽过滤 + 增加结果数 (论文 141→263)

**依赖**：config.py, models.py

**建设顺序**：第 3 个文件。

---

### 4. `downloader.py` — PDF 解析 + 语义分块

**功能**：PyMuPDF 提取 PDF 全文 + 图片，文本分块。

**核心函数**：
- `parse_pdf(pdf_bytes, pdf_name)` — 提取全文 + 图片
- `chunk_text(text, chunk_size)` — 固定大小滑窗分块
- `semantic_chunk_text(text, max_chars, overlap)` — **语义分块**（句子边界 + 重叠窗口）

**优化历程**：
1. 最初只有固定 2048 字符滑窗
2. 加入 `semantic_chunk_text` — 按句子边界切分，2 句重叠，保留语义完整性

**依赖**：config.py

**建设顺序**：第 4 个文件。

---

### 5. `knowledge_base.py` — LanceDB 知识库

**功能**：双层向量索引构建 + 检索。

**核心函数**：
- `_get_embed_model()` — bge-small-zh 模型加载
- `_embed_texts_sync(texts)` — 批量向量化
- `_get_table(db, table_name)` — 获取/创建 LanceDB 表
- `build_knowledge_base(papers, chunks, topic_id)` — 构建知识库 (摘要层 + 全文层)
- `store_image_descriptions(table_name, records)` — 图片描述存储
- `_hybrid_search(table, q_str, q_vec, top_k, flt)` — BM25 + 向量混合检索 (RRF 融合)
- `retrieve_with_images(table_name, query, top_k)` — 三层检索 (文本 + 图片 + 摘要)
- `_cosine_search(table, query_vec, top_k, filter_expr)` — 纯向量检索

**表结构**：
```
chunk_id | paper_id | title | authors | year | abstract | doi
is_fulltext | is_image | image_path | page_number | text | vector(512)
```

**优化历程**：
1. 最初使用 Jina AI Embeddings API → 限速 15 RPM → 切换到本地 bge-small-zh
2. 增加 `is_image` 字段支持图片描述存储
3. 增加 `_hybrid_search` — BM25 + 向量 RRF 融合
4. 增加 FTS 全文索引支持

**依赖**：config.py, models.py

**建设顺序**：第 5 个文件。

---

### 6. `qa_engine.py` — RAG 问答引擎 (核心)

**功能**：完整的 RAG 链路——查询改写 → 混合检索 → 级联重排 → LLM 生成。

**核心函数**：
- `_expand_query(question)` — DeepSeek 查询改写
- `_validate_expansion(original, expanded)` — **语义校验**（改写与原问题余弦相似度 ≥ 0.75）
- `_is_complex_query(question)` — **策略路由**（复杂查询才触发 Rerank）
- `_rerank(question, chunks, top_k)` — bge-reranker-base CrossEncoder 重排序
- `_get_reranker()` — 延迟加载 Rerank 模型
- `ask_question(table_name, question)` — 主问答入口
- `ask_with_context(table_name, question)` — 返回上下文版本（供评估用）

**RAG 链路流程**：
```
用户问题
  → _expand_query (DeepSeek 改写)
  → _validate_expansion (语义校验, 低于 0.75 回退)
  → HybridRetriever.search (自适应 RRF + 级联重排)
  → _is_complex_query 判断 (策略路由)
  → DeepSeek 生成 (带多轮记忆 + 引用 + 补充线索)
```

**优化历程**：
1. 最初只有向量检索 + Prompt 生成
2. 加入 Rerank 重排序 (bge-reranker-large → bge-reranker-base)
3. 加入查询改写 + 语义校验
4. 加入策略路由 (复杂/简单查询分流)
5. 集成 HybridRetriever (自适应 RRF + 级联重排)
6. 集成 ConversationMemory (多轮记忆)

**依赖**：config.py, models.py, knowledge_base.py, hybrid_retriever.py, conversation_memory.py

**建设顺序**：第 6 个文件。

---

### 7. `hybrid_retriever.py` — 自适应混合检索器 (自研核心)

**功能**：本项目最核心的技术壁垒——不是调 LangChain 的包，而是自研的检索算法。

**核心类**：`AdaptiveHybridRetriever`

**核心方法**：
- `_compute_adaptive_weight(query)` — **4 维特征推理 BM25 权重** (查询长度/英文术语/问句形式/长文本)
- `_vector_recall(query, top_k)` — bge 向量检索
- `_bm25_recall(query, top_k)` — LanceDB FTS BM25 检索
- `_adaptive_rrf(vec, bm25, query, top_k)` — **自适应 RRF 融合** (动态权重, 非固定 50:50)
- `_cascade_rerank(query, chunks, top_k)` — **两阶段级联重排** (Stage1 MiniLM 预筛 → Stage2 bge-reranker 精排)
- `search(query, top_k, filter_expr)` — 主入口

**与开源方案的区别**：
- LlamaIndex/LangChain: 固定权重混合检索
- 本项目: 根据问题特征动态调整 α = f(长度, 术语, 句式, 复杂度)
- 输出 alpha 值供消融实验可解释

**依赖**：knowledge_base.py, sentence_transformers

**建设顺序**：后期文件（MVP 完成后的深度优化阶段）

## v1.2 更新 (2024-06-21)：三项 RAG 优化 + GitHub 发布

### 1. 多轮指代消解 (Contextualized Query Rewriting)

**问题**：用户第二轮问"它有什么缺点"时，检索 query 是"它有什么缺点"而非"深度学习有什么缺点"。

**方案**：
- `qa_engine.py` 新增 `_resolve_coreference(question, history)` → DeepSeek 将"它/这/它们"消解为具体名词
- 仅在对话历史存在 + 问题 < 20 字时触发
- `COREF_RESOLVE_PROMPT`: temperature=0.1, max_tokens=100, 超时 8s
- 降级策略：LLM 失败用原问题

**测试**：5/5 指代消解通过
| 历史 | 问题 | 消解结果 |
|------|------|---------|
| 什么是深度学习 | 它有什么缺点 | 深度学习有什么缺点 ✅ |
| RAG是什么 | 这种方法有什么优势 | RAG这种方法有什么优势 ✅ |
| Transformer架构 | 它的注意力机制如何工作 | Transformer架构中的注意力机制如何工作 ✅ |

### 2. 父子分块 (Parent-Child Chunking)

**问题**：检索到的小 chunk 缺少上下文（如缺表头的表格片段），LLM 无法正确理解。

**方案**：
- `knowledge_base.py` schema 新增 `chunk_index` 列 (abstract=0, fulltext=1+)
- `qa_engine.py` 检索后查 LanceDB 同论文 chunk_index±1 的邻居块 → 追加到结果
- 降级策略：表无 chunk_index 列时静默跳过

### 3. Agent 自反思 (Self-Reflection)

**问题**：Agent 生成答案后没有检查步骤，可能遗漏重要方面。

**方案**：
- `agent/agent_core.py` 在 LLM 返回最终答案后追加反思步
- 反思 Prompt："是否遗漏了重要方面？是否引用了足够的论文依据？"
- temperature=0.2, max_tokens=600
- 反思结果 > 20 字则替换原答案
- 仅当剩余步数 > 0 时触发
